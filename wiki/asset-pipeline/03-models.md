# XModels (3D models)

> *In this project, xmodel work is ~95% placement (in Radiant) and ~5% GDT editing (in APE). The page is weighted accordingly.*

## Mental model

Three layered concepts. Don't conflate.

- **Source mesh**: a `.XMODEL_EXPORT` (ASCII text) or `.XMODEL_BIN` (binary) file on disk — the geometry + UVs + skeleton + per-vertex weights authored by an artist or extracted from a source game. Located under `model_export/` (or any root listed in `bin/converter_gdt_dirs_0.txt`).

  > 💡 `.XMODEL_EXPORT` really is ASCII — open one in a text editor and you'll see a header like `// Generated by Wraith - Game extraction tools` followed by `MODEL`, then verbose blocks of vertices, faces, and bone data. That's surprisingly handy: you can grep an export for material slot names, diff two exports, or hand-patch one in a pinch. The `.XMODEL_BIN` is the compiled binary the engine actually consumes — same data, packed.
- **`xmodel` GDT record**: declares the asset to the engine — names it, points at the source mesh, configures LODs, collision, physics, and per-slot material assignments.
- **Placement**: a Radiant *entity* with `model` set to the xmodel asset name (or in a few cases, the model swapped at runtime from script via `SetModel()`).

```
.XMODEL_BIN on disk   ──►   xmodel GDT record (in APE)   ──►   Radiant entity with `model "<name>"`   ──►   in-game
```

The Radiant brush face you painted with a material (page 02) and the entity you placed referencing an xmodel are *different rendering paths*. Brush faces become part of the BSP. Models are placed objects with their own draw calls and culling.

## Where models come from

Same three-flow story as images-and-materials. In this project:

| Origin                                                                       | Frequency in `zm_test`         |
| ---------------------------------------------------------------------------- | ------------------------------ |
| **Pre-extracted community packs** (Midgetblaster, harrybo21, KingslayerKyle, SkyeLord, MoiCestTOM…) | Dominant. Pack ships the `.XMODEL_BIN` files + a GDT declaring them. |
| **Greyhound / Saluki extraction by you**                                     | Rare — most porting was already done by pack authors.                                                  |
| **Authored** (Maya / Blender → CoDMayaToolsX or similar exporter → `.XMODEL_EXPORT`) | None in this project — Reapy is not a 3D modeller. Mentioned only for completeness. |

The install pattern mirrors images: drop the binaries into `model_export/_<author>/...`, drop the GDT into `source_data/`, fix paths if the pack expects a different layout, then reference / zone.

## The `xmodel` GDT record

~50 meaningful fields. The semantically interesting ones cluster.

### 1. The source mesh

- **`filename`** — relative path to the LOD0 `.XMODEL_BIN` file. Resolved against `converter_gdt_dirs_0.txt` roots. Example: `_bo6\\Foliage\\<...>\\<name>_LOD0.XMODEL_BIN`.
  > Don't be fooled by the `_LOD0` suffix in the path — that's just naming convention for the highest-detail mesh. The other LODs are either (a) auto-generated by the engine via the `autogenLod*` flags or (b) separate `.XMODEL_BIN` files referenced by `filenameLOD<N>` fields.

### 2. Type & physics

- **`type`** — what kind of model this is. APE's dropdown values:
  - `rigid` — static mesh, no skeleton. Walls, props, foliage. The vast majority.
  - `animated` — has a skeleton + bone weights. Characters, dynamic props, animated objects.
  - `multiplayer body` — MP-specific player body rig.
  - `viewhands` — first-person view-model arms.
- **`physicsConstraints`** — links to a `physconstraints` asset for ragdoll/jointed-physics models.
- **`physPreset`** — links to a `physpreset` asset for `dyn_model` collision/mass/friction.
- **`BulletCollisionFile`** / **`BulletCollisionLOD`** / **`BulletCollisionRigid`** — separate per-LOD collision mesh for bullets (cheap collision approximation distinct from the render mesh).
- **`CollisionMap`** — link to a `col_map` asset for player/AI collision.

### 3. LODs

LODs (Levels of Detail) swap to lower-poly meshes as the model moves further from the camera, saving rendering cost.

- **`autogenLod4` … `autogenLod7`**, **`autogenLowLod`**, **`autogenMediumLod`**, **`autogenLowestLod`** — toggles for engine-side automatic LOD generation. With these on, the engine builds simplified meshes from the LOD0 source.
- **`autogenLod4Percent` … `autogenLowestLodPercent`** — target poly-count percentage for each auto-generated tier (e.g. `LowLodPercent "25"` = 25% of LOD0's polys).
- **`highLodDist`**, **`mediumLodDist`**, **`lowLodDist`**, **`lowestLodDist`** — distance thresholds (in inches) at which each LOD swaps in. `0` = use engine defaults.
- **`dropLOD`** — `Auto` lets the engine cull LODs that wouldn't be useful; manual values override.
- **`customAutogenParams`** — `1` to override the auto-gen defaults with the per-LOD percentages above; `0` to use engine defaults.

> 💡 **In this project, LODs were manually tweaked on only ~3 or 4 models.** The default auto-gen settings shipped by community packs are usually fine. Only touch these if you observe specific perf problems (a model still high-poly far away) or visual problems (a low-LOD swap causing visible popping). Verify with the `assetinfo/zm_test_xmodel.csv` report.

### 4. Material assignment

The xmodel's source mesh defines named *slots* (per-mesh-section material names baked in by the modelling tool). The GDT exposes two ways to override what gets bound to those slots, without re-exporting the mesh:

- **`materials`** — a single field on the xmodel record holding a space-separated list of material names that override the mesh's baked-in defaults. Quick and inline; suitable for simple cases.
- **`skinOverride`** — links to a separate **skin** asset. A skin is its own GDT construct that maps mesh-slots → materials in a more structured way; useful when you want multiple skins for one mesh (e.g. clean / dirty / camo variants of the same model) and to swap them at runtime.

If both fields are blank, the slot uses the material name baked into the source mesh.

### 5. Visual / culling

- **`cullOutDiameter`** — model culls beyond this distance regardless of LOD setup. `0` = no extra cull.
- **`cullOutOffsetCP`** / **`cullOutOffsetMP`** — per-game-mode (campaign / MP) cull tuning.
- **`DetailShadows`** — `1` = include in detail shadow pass.

### 6. Bones & rigging (for animated models)

- **`boneControllers`** — names of bones that get procedural control (e.g. eye-aim, recoil-driven offsets).
- **`boneStabilizers`** — bones that get stabilization passes (e.g. spine counter-rotation).
- **`skinXModel`** — link to a paired skinning xmodel for character bodies.

You almost never edit these for ported assets. They're set by the original author.

## Placement in Radiant

This is where you'll spend 95% of your xmodel time. There's a small palette of entity classes, each with a distinct role.

| Entity classname     | What it is                                                                            | When to use                                                                            |
| -------------------- | ------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| `misc_model`         | **Static placed model.** Baked into the BSP at compile time — no runtime entity, just geometry painted into the world. | Every "decorative" prop that never moves. The vast majority of foliage, furniture, signage, etc. Cheap. |
| `script_model`       | **Scripted entity** holding a model. Has a runtime entity reference; can be moved, rotated, hidden, swapped via GSC. | Anything you'll touch from script — interactives, animated props, things that appear/disappear during gameplay. |
| `script_brushmodel`  | **Scripted brush entity.** A brush group treated as one entity with collision; can be moved/rotated. | Movable doors, elevators, sliding panels. The model is the brush geometry itself.       |
| `dyn_model`          | **Dynamic-physics model.** Has full physics simulation — pushable, falls under gravity, ragdolls.       | Pushable barrels, tipping debris, anything you want the player to be able to knock around. |

### Which one for which job?

- "I want this model here, doesn't move, doesn't talk back" → **`misc_model`**.
- "I want to hide/show this from script" or "I want to play an animation on it" → **`script_model`**.
- "I want this door to swing open" → **`script_brushmodel`** (the door's brush geometry as the entity).
- "I want this barrel to fall over when shot" → **`dyn_model`**.

In `zm_test`, the top-level `zm_test.map` itself only places a handful of model entities — most placement is *inside prefabs* (`_prefabs/zm/zm_test/house/*.map`), which then get instanced multiple times into the world. That's why the `map_stats` report counts model placements per prefab × times-included.

### Tips

- **Group repeated decorative models into a prefab.** It's the only way to get clean re-use without 200 hand-placed entities cluttering the brush list. The `house_balcony_panel_single.map` prefab is included **39 times** in zm_test — one source, 39 placements.
- **Snap to grid** before placement to avoid Z-fighting against brushes. Radiant defaults are usually fine.
- **Set `targetname`** on `script_model`s so GSC can find them — `GetEnt("my_model_tag", "targetname")`.

## Worked example: adding a new prop to your map

1. **Get the `.XMODEL_BIN`** — install a community pack, or extract via Greyhound/Saluki and copy into `model_export/<your-folder>/`.
2. **GDT entry** — usually shipped with the pack. If you authored: create an `xmodel` record in APE pointing at your `.XMODEL_BIN`, set `type` (likely `rigid`), accept default LOD auto-gen.
3. **Place in Radiant** — open the level, create an entity (right-click → entity → `misc_model`), set its `model` property to your xmodel asset name, position it, snap to grid.
4. **Zone if needed** — if you placed it via `misc_model` on the BSP, the BSP carries it. If you only swap to it from script, add `xmodel,<name>` to your `.zone`.
5. **Compile (All if structural changes, Just Ents if you only added entities) → Light → Link → Run.**

For a `script_model` you'll also need to give it a `targetname` and write the GSC that hooks onto it.

## Common gotchas

- **Material is purple-checker / black on the model.** The model's slot material is missing — either not zoned, not in any GDT, or bound to a name that doesn't exist. Check the slot binding (mesh-baked or GDT-overridden), then check the material exists.
- **Visible LOD popping** when moving the camera. `mediumLodDist` / `lowLodDist` set too aggressively, or auto-gen LOD percentages too low. Either bump distances out or tune percentages.
- **Performance dies in a zone with lots of foliage.** Auto-gen LODs not configured (everything stays at LOD0). Verify in `assetinfo/zm_test_poolinfo.csv` and the `map_stats` per-prefab model count.
- **Pathing breaks around a placed model.** AI nav-mesh doesn't see model geometry by default — wrap in a clip brush (or use `script_brushmodel` for big movable obstacles) so AI knows to route around it.

> A model "missing" in-game is *almost always a mapping problem* (umbra culling cell misconfigured, the model is outside any visible portal-cell) rather than an asset-pipeline problem. See the mapping doc when we get to umbra/portals.
