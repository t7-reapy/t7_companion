# FX recipes — playing FX from script

> Cookbook of FX patterns: register an effect, play it on the world / a tag / the camera, manage its lifecycle, swap GSC ↔ CSC for the right side. Anchored in `zm_test`'s 221 `level._effect[...]` usages and cross-checked against the GSCode VS Code extension (see [`api-reference.md`](./api-reference.md)).

## Mental model: register, then play

Playing an FX in BO3 from script is a **two-step** thing (with a script-free third path via Radiant — see below):

1. **Register** — at script init, store the FX's *relative path from the BO3 root* in `level._effect[name]`. The string is just a path under `share/raw/fx/...` (with the `share/raw/fx/` prefix elided in the value — you write `"zombie/fx_powerup_grab_solo_zmb"`, not the absolute path). Registration is per-side: assign on GSC if you'll play it from server, on CSC if you'll play it from client, on both if you need it on both.
2. **Play** — call `PlayFX(...)` / `PlayFxOnTag(...)` / etc. with `level._effect[name]` as the first argument. The play call dispatches the effect at a position / on a tag / on the camera.

Plus a **`#precache` directive** at the top of the file so the linker bundles the FX into the fastfile (see overview [`01-overview.md`](./01-overview.md#2-asset-precaching--why-did-my-model-lag-the-first-time-it-appeared)). **Precaching is mandatory for FX** — without it, the play call won't find the effect at runtime. (More gotchas about precaching limits below.) And of course: the FX itself has to be **zoned** so the linker actually includes its `.efx` in your fastfile.

```
#precache directive   ──►  declares "I'll need this FX, bundle it in"
.efx zoned in .zone   ──►  the actual FX file ends up in the fastfile
level._effect[name]   ──►  builds your runtime name → path lookup
PlayFX*(...)          ──►  spawns the effect in-game
```

> 💡 **One-step alternative**: drop an FX entity directly into the map in Radiant. No script involvement, no `level._effect[]`, no `PlayFX(...)` call — Radiant's exploder / FX entities fire on map start (or via the lighting-state mechanism, see [`asset-pipeline/05-ssi.md`](../asset-pipeline/05-ssi.md)). This is the right path for static / ambient FX (drainpipe drips, candles, idle particles). The script-side workflow above is for *dynamic* FX driven by gameplay events.

## The register step

Three patterns, all in `zm_test`:

```c
// 1) Inline literal path (server side, simplest case)
//    on disk: share/raw/fx/zombie/fx_powerup_grab_solo_zmb.efx
level._effect["powerup_grabbed"] = "zombie/fx_powerup_grab_solo_zmb";

// 2) Path constant from a .gsh header (project-wide reuse)
//    FX_POWERUP_BLUE expands to a path string defined in a shared .gsh
level._effect["powerup_on"] = FX_POWERUP_BLUE;

// 3) Computed / templated path (XcDylan93-style advanced)
//    on disk: share/raw/fx/_xcdylan93/<color>/muzzle/fx_muzflash_<color>.efx
level._effect["muz_flash"] = "_xcdylan93/" + color + "/muzzle/fx_muzflash_" + color;
```

Pattern 3 is what powers XcDylan93's colour system in [`ReDempTion/scripts/zm/_xcdylan93/_zm_mod_fx.csc`](https://github.com/XcDylan932/ReDempTion/blob/main/scripts/zm/_xcdylan93/_zm_mod_fx.csc) — register N effects across M colours by iterating `array("red","blue",...)` × category names. Worth reading the whole file if you want to push beyond basic usage.

`level._effect[]` is **populated independently on GSC and CSC sides** — script writers assign to the same key on both sides where each side needs it. The dictionary itself lives in the runtime per side.

## The play API — GSC vs CSC

> ℹ️ **Capitalisation doesn't matter — function names are fully case-insensitive on both sides.** The GSCode JSON happens to record GSC entries as `PlayFX*` (uppercase X) and CSC entries as `PlayFx*` (lowercase x), but you can write either spelling on either side and the engine will resolve it. The names below use the canonical case from the JSON purely for consistency with what hover docs will show you.

| What you want                              | GSC (server)                           | CSC (client)                                  |
| ------------------------------------------ | -------------------------------------- | --------------------------------------------- |
| Spawn FX at world position                 | `PlayFX(fx, pos, fwd, up, ignorePause)` | `PlayFx(localClientNum, fx, pos, fwd, up, time, primLightFrac, lightOriginOffset)` |
| Spawn FX on a model tag (bone/socket)      | `PlayFXOnTag(fx, ent, tag, ignorePause)` | `PlayFxOnTag(localClientNum, fx, ent, tag)` |
| Spawn FX on the local player's camera      | `PlayFXOnCamera(fx, offset, fwd, up, ignorePause)` — single shipped example: `t7-source/scripts/cp/cp_prologue_intro.gsc:96`. Rare in practice; `PlayFxOnCamera` (CSC) is the conventional path. | `PlayFxOnCamera(localClientNum, fx, offset, normal, forward)` |
| Spawn FX on a dynamic entity               | —                                      | `PlayFXOnDynEnt(...)`                          |
| Spawn FX on a viewmodel                    | —                                      | `PlayViewmodelFx(...)`                         |
| Spawn a `client_tagfxset` (multi-tag set)  | —                                      | `PlayTagFXSet(...)`                            |
| Check if an FX is still playing            | —                                      | `IsFXPlaying(effectId)`                        |
| Stop a previously-played FX                | —                                      | `StopFX(localClientNum, effectId)`             |

**Key signatures explained:**

- **`PlayFx` (CSC)**: 8 parameters, last 3 optional (`time`, `primLightFrac`, `lightOriginOffset`). "Playback" here means the FX runtime: `time` overrides the FX's natural duration, `primLightFrac` overrides the primary-light influence (for self-illuminating FX like sparks), `lightOriginOffset` shifts where the FX's emitted light originates from (vs the FX's own origin).
- **`PlayFxOnCamera` (CSC)** uses the parameter name `normal` (not `up`) for the second-to-last vector — same role, just a different convention than `PlayFx`'s `up` argument.
- **`IsFXPlaying` / `StopFX` work in CSC only.** They operate on an **effect id** (an integer) — the return value of CSC `PlayFx*` calls. Stash the return when you spawn the FX, then pass it later to `IsFXPlaying` / `StopFX`.

> ⚠️ **GSC has no effect-id concept.** GSC `PlayFX*` calls don't return a manageable handle. To stop a long-running FX from GSC, you typically spawn a **script_origin** or **script_model** entity, attach the FX to it via `PlayFXOnTag`, and **`Delete()` the entity** when you want the FX gone. Killing the entity kills the attached FX. (See examples below.)

The CSC API is broader because **per-client visual state lives on the client** — the server logic is shared across all players (it's authoritative game state), while each player's screen / viewmodel / camera / per-client dynamic effects live in CSC. Anything camera/viewmodel/dynamic-entity-related is CSC-only.

**The first `localClientNum` argument on every CSC call** is the splitscreen client index. You almost always pass the `n_client_num` you got from the surrounding clientfield/callback handler. In a `foreach (player in GetLocalPlayers())` loop, use `player GetLocalClientNumber()`.

## Recipes

### Basic GSC spawn at a world position (one-shot)

```c
// register at init
level._effect["switch_sparks"] = "switches/fx_switch_sparks";

// later
PlayFX(level._effect["switch_sparks"], spark_fx.origin, forward);
```

This is fire-and-forget — once the FX finishes its natural duration, it cleans itself up. Nothing to stop.

### Stoppable FX from GSC — entity-anchored

When you need the FX to keep playing until you decide to stop it, the GSC pattern is to **anchor it to an entity** and `Delete()` the entity to stop:

```c
// Spawn an invisible entity at the position, attach FX to it
fx_anchor = Spawn("script_origin", origin);
PlayFXOnTag(level._effect["my_loop_fx"], fx_anchor, "tag_origin");

// ... later, when you want to stop:
fx_anchor Delete();   // attached FX dies with the entity
```

For `script_model` entities (visible models with FX attached), the same pattern applies — `Delete()` the model and the FX goes with it.

### CSC spawn at a position with explicit client number, stoppable

From your cellbreaker AI:

```c
PlayFx(localClientNum, "custom/AI/cellbreaker_spawn", self.origin, AnglesToForward(self.angles));
```

For a stoppable CSC FX, capture the returned effect id:

```c
my_fx_id = PlayFx(localClientNum, level._effect["my_fx"], origin, forward, up);

// ... later, optionally check it's still alive:
if (IsFXPlaying(my_fx_id))
{
    StopFX(localClientNum, my_fx_id);
}
```

### Attach FX to an entity tag (GSC)

```c
PlayFXOnTag(level._effect["wolf_bite_blood"], self, "tag_mouth_fx");
```

`self` is the entity (zombie, weapon, prop). `"tag_mouth_fx"` is a bone/tag name on the model.

### Attach FX to a tag, holding onto the handle (CSC, looping)

The apothicon-fury archetype keeps an array of looping FX handles for later cleanup:

```c
self.loopingEffects[0] = PlayFXOnTag(localClientNum, level._effect[FURY_BREATH_EFFECT],     self, "j_head");
self.loopingEffects[1] = PlayFXOnTag(localClientNum, level._effect[FURY_BODY_SMOKE_EFFECT], self, "j_spine4");
self.loopingEffects[2] = PlayFXOnTag(localClientNum, level._effect[FURY_FOOTSTEP_AMB_EFFECT], self, "j_ball_le");
// ... etc
```

(See the **stoppable CSC** recipe above for how to clean these up via `StopFX(localClientNum, effectId)`.)

### Camera-attached FX (CSC only — camera lives on the client)

`zm_test`'s hellround transition uses this for the bloody-screen FX as the player crosses into the bloody round:

```c
PlayFXOnCamera(n_client_num, level._effect["parasite_round"]);
```


### Server-triggered, client-played (the canonical pattern for gameplay-driven visual FX)

Most FX you fire from gameplay logic want to **start on the server** (so it's authoritative, all clients see it) but **render on the client**. The bridge is a clientfield — the typed bit-channel from server to clients. Full protocol in `clientfields.md` *(planned)*; here's the FX-specific shape end-to-end:

**1. Define the clientfield in a `.gsh` so both sides see the name:**

```c
// in scripts/zm/my_module/my_module.gsh
#define MY_FX_CF_NAME "play_my_fx"
```

**2. CSC side — `#precache` the FX, register the clientfield, write the handler:**

```c
#precache("client_fx", "myproject/fx_my_effect");

REGISTER_SYSTEM("my_module", &init, undefined)

function init()
{
    level._effect["my_fx"] = "myproject/fx_my_effect";

    // Register a counter-style clientfield on the scriptmover entity type.
    // CF_CALLBACK_ZERO_ON_NEW_ENT fires the handler on late-joiners too.
    clientfield::register("scriptmover", MY_FX_CF_NAME, VERSION_SHIP, 1, "counter",
                          &on_play_my_fx, !CF_HOST_ONLY, CF_CALLBACK_ZERO_ON_NEW_ENT);
}

function on_play_my_fx(localClientNum, oldVal, newVal, bNewEnt, _bInitialSnap, _fieldName, _bWasTimeJump)
{
    util::waitforclient(localClientNum);
    PlayFxOnTag(localClientNum, level._effect["my_fx"], self, "tag_origin");
}
```

**3. GSC side — `#precache` (cross-VM!), register, then trigger by incrementing:**

```c
#precache("fx", "myproject/fx_my_effect");   // independent precache for the server VM

REGISTER_SYSTEM("my_module", &init, undefined)

function init()
{
    level._effect["my_fx"] = "myproject/fx_my_effect";

    // Same clientfield must be registered on the GSC side too.
    clientfield::register("scriptmover", MY_FX_CF_NAME, VERSION_SHIP, 1, "counter");
}

// Somewhere in gameplay logic, on the entity you registered the field on:
function trigger_fx_on(target_ent)
{
    target_ent clientfield::increment(MY_FX_CF_NAME);
}
```

The clientfield's `"counter"` semantics + `increment()` make it a **fire-and-forget pulse** — every increment fires the CSC handler once on every client. This is the right shape for one-shot FX events. For continuous on/off state (like a glow that toggles), use `"int"` with bit width 1 and `set()`/`clear()` instead of increment.

## Stopping / killing FX — summary

- **GSC**: spawn a `script_origin` / `script_model`, attach FX with `PlayFXOnTag`, `Delete()` the entity to stop. (See "Stoppable FX from GSC" recipe above.)
- **CSC**: keep the effect id from `PlayFx*`, then `StopFX(localClientNum, effectId)`. (See the stoppable-CSC recipe.)
- **Radiant exploders**: use `exploder::kill_exploder(name)` from the utility library — see next section.

## Radiant exploders — FX placed in the map editor

Radiant lets you drop "exploder" entities into the map, each with a name (or numeric ID) and one or more attached FX.

**Always go through the utility library** at `share/raw/scripts/shared/exploder_shared.{gsc,csc}` rather than the raw `PlayRadiantExploder` / `StopRadiantExploder` / `KillRadiantExploder` engine APIs. The wrapper handles edge cases (multi-side firing, kill-then-spawn timing, reference counting) that the raw APIs don't:

```c
#using scripts\shared\exploder_shared;

// Fire an exploder by name
exploder::exploder("teleporter_controller_light_1");

// Stop / kill an exploder by name
exploder::kill_exploder("teleporter_controller_red_light_1");
```

(Real example from `share/raw/scripts/zm/zm_giant_teleporter.gsc:96–102`.)

`zm_test` uses exploders heavily for weather (rain pipes, wind bursts) and hellround environment FX. They're often triggered indirectly via the **lighting-state mechanism** — FX entities have `lightingstate_N` flags so they auto-toggle with state changes (see [`asset-pipeline/05-ssi.md`](../asset-pipeline/05-ssi.md)) — rather than direct script `exploder::*` calls.

  ```c
  #precache("client_tagfxset", "ability_hero_heat_wave_player_impact");
  // ...
  playtagfxset(localclientnum, "ability_hero_heat_wave_player_impact", self);
  // signature: PlayTagFXSet(localClientNum, tagFxSetName, entity)
  ```

  The TagFXSet itself is authored separately (it bundles multiple FX/tag pairs into one named set). Used only on a few abilities/gadgets in shipped Treyarch — see also `_gadget_flashback.csc` and `_gadget_heat_wave.csc` for more.
- **`PlayViewmodelFx`** — for viewmodel-specific FX (muzzle, ejection, custom weapon events).

## Common pitfalls

- **FX won't play at all** — by far the most common cause is missing the `#precache` directive. Without it, the FX simply doesn't fire (no hitch, no error, no playback). Add `#precache("fx", ...)` in GSC or `#precache("client_fx", ...)` in CSC.
- **FX is on the wrong side.** `level._effect[name]` was set on GSC but the play call is in CSC (or vice versa). Both VMs maintain independent dictionaries — assign on the side you'll play from.
- **CSC FX appears once but never again** — the effect-id reference was lost. Stash the id when you spawn, `StopFX(localClientNum, effectId)` when done.
- **Camera FX on a wrong client number in splitscreen** — only player 1 sees it. The `localClientNum` argument is mandatory and side-specific; pass the right one per local client.
- **Exploders state mismatch** — Radiant exploders are configurable (default-on / default-off / triggered-by-script). The common bug is configuring the *exploder state* opposite to the *FX entity's default state*, so the FX plays when the exploder is "off" or vice versa. See the dedicated exploders page (planned) for the wiring rules.
- **Hitting BGCache caps** — adding more `#precache` entries than the cap (256 for `serverfx`, 1024 for `clientfx`, etc.) causes link-time errors. See [`docs/reference/bgcache-caps.md`](../reference/bgcache-caps.md) for the full table and verification paths.

### Precache limits — the BGCache caps

FX precaches go into the **BGCache** ("Both Game" cache — a project-wide cache distinct from server-only or client-only caches). Each precache type has a hard ceiling. The two FX-relevant ones:

- **`serverfx` (GSC `#precache("fx", ...)` declarations): cap = 256.**
- **`clientfx` (CSC `#precache("client_fx", ...)` declarations): cap = 1024.**

Verified caps confirmed against community-reported numbers and in-engine error behaviour. **For the full BGCache cap table** across all asset types (models, weapons, materials, strings, menus, etc.), see [`docs/reference/bgcache-caps.md`](../reference/bgcache-caps.md).

Other rules:

- **The two VMs don't share precaches.** Precaching an FX in GSC does **not** make it available to CSC, and vice versa — server and client VMs are independent. You must precache on each side that uses the FX.
- **Radiant-placed FX still need a script `#precache` if you'll reference them from script.** Some Radiant FX work without explicit precache because *another script* already precached them transitively — that's why the same FX can mysteriously work in one map and fail in another.

**Verification paths:**

- Run **`/listassetpool 0-102`** in the in-game console to dump current pool counts (one row per asset-type index). Sample row format:

  ```
  Total of 1499/2000 assets in fx pool, bytes 215856
  ```

  (Pass narrower ranges like `/listassetpool 38` if you only care about a specific type — see [`docs/reference/runtime-asset-pools.md`](../reference/runtime-asset-pools.md) for the complete index → name → cap mapping.)
- The `-printbgcache` modtools flag *should* dump BGCache contents at link time, but it may need to be passed manually to `linker_modtools.exe` or `modtools.exe` rather than the launcher GUI. Behaviour not fully verified. (Full args list in [`docs/reference/modtools-cli-args.md`](../reference/modtools-cli-args.md).)
- Open `usermaps/zm_test/zone_source/all/assetinfo/zm_test_poolinfo.csv` (or its `english/` / `french/` siblings) — the linker emits a `type,limit,total` row per asset type. **Verified contents** (excerpt):

  ```
  type,limit,total
  fx,1875,0
  physpreset,276,1
  physconstraints,128,0
  destructibledef,128,0
  xanim,25000,0
  ```

  Note: `fx` here (`1875`) is the **engine runtime asset pool** — a different concept from the `serverfx`/`clientfx` BGCache caps above. Both are real ceilings, but the BGCache cap (256/1024) is the one you trip first in practice. (Reports documented in [`asset-pipeline/01-overview.md`](../asset-pipeline/01-overview.md#generated-reports--diagnostic-files).)

> If you start hitting "Could not find fx" errors after adding a new module, **suspect zoning first** — the FX simply isn't in the fastfile. Add the right `fx,<path>` / `fx,<path>` line in your zone file. The other causes (BGCache cap exceeded, wrong side) are downstream and only matter once zoning is correct.
